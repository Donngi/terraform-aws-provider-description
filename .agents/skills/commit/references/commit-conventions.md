# コミットメッセージ規約

[Conventional Commits](https://www.conventionalcommits.org/ja/v1.0.0/)を基本とした規約。

## フォーマット

```
<type>(<scope>): <subject>

<body>

[optional footer(s)]
```

## Type（変更種別）- 必須

| Type | 用途 | 例 |
|------|------|-----|
| `feat` | 新機能追加 | 新しいエンティティ、API、UI機能 |
| `fix` | バグ修正 | 既存機能の不具合修正 |
| `refactor` | リファクタリング | 動作を変えないコード改善（**振る舞いが変わる場合は`feat`か`fix`**） |
| `docs` | ドキュメント | README、コメントのみの変更 |
| `test` | テスト | テストコードの追加・修正 |
| `chore` | 雑務 | 依存関係更新、設定変更、CI/CD |
| `perf` | パフォーマンス改善 | 処理速度、メモリ使用量の最適化 |
| `style` | スタイル | フォーマット、lint修正（動作変更なし） |

### Type選択ガイドライン

| 状況 | Type |
|------|------|
| 新しいファイルを追加して機能を実装 | `feat` |
| 既存機能の不具合を直す | `fix` |
| 動作は変わらないがコードを整理 | `refactor` |
| 既存の動作を変更・拡張する | `feat`（新機能）または `fix`（修正） |
| テストコードのみの変更 | `test` |
| package.jsonの依存関係更新 | `chore` |
| CI/CD設定の変更 | `chore` |
| READMEの更新 | `docs` |
| 処理速度を向上させる変更 | `perf` |
| インデントやフォーマットのみ | `style` |

## Scope（影響範囲）- 必須

変更の影響範囲を示す。プロジェクトの構造に応じて決定。

**指針**:
- パッケージ名、モジュール名、機能名などを使用
- 複数パッケージにまたがる場合は、最も重要な変更があるパッケージをスコープにする
- 本当に同等の重要度の場合のみ `global` を使用
- コミットが大きすぎる場合は分割を検討

```
# 良い例: 主パッケージを明示
feat(auth): ログイン機能を追加
fix(api): エラーハンドリングを修正

# 避けたい例: 何でもglobal
feat(global): 認証機能を追加
```

## Subject（件名）- 必須

- **日本語で記述**する
- **動詞で始める**（「追加」「修正」「変更」「削除」「改善」など）
- **50文字以内**を目安とする
- 末尾に句点（。）は付けない

## Body（本文）- 推奨

**「なぜこの変更をしたのか」** を記述。

### 記述すべき内容

- **変更の動機・背景**: なぜこの変更が必要だったのか
- **選択した理由**: 複数の選択肢がある中でなぜこの方法を選んだのか
- **トレードオフ**: この変更によって得られるものと失われるもの

### 記述のポイント

- 件名から1行空けて記述
- **何を**変更したかだけでなく、**なぜ**変更したかを説明

## コミット分割の指針

### 1コミットの目安

- 1つの論理的な変更
- レビュー可能なサイズ（差分300行以内が目安）

### 分割すべき場合

- 異なる機能の変更が混在している
- 差分が大きすぎてレビューしづらい
- `feat` と `fix` が混在している
- 独立してレビュー・リバート可能な変更が含まれている

### 分割の例

```
# 悪い例: 1コミットに詰め込みすぎ
feat(global): ユーザー機能とバグ修正

# 良い例: 論理単位で分割
feat(domain): Userエンティティを追加
feat(api): ユーザー登録APIを実装
feat(ui): ユーザー登録フォームを追加
fix(ui): ボタンのスタイル崩れを修正
```

## コミットメッセージの例

### 新機能追加

```
feat(domain): UserSaveStatusエンティティを追加

ユーザーが記事を「後で読む」として保存できる機能の基盤として、
保存状態を管理するエンティティを追加。
既読状態とは独立して管理することで、
「読んだけど保存しておきたい」ユースケースにも対応可能。
```

### バグ修正

```
fix(ui): 未読フィルター適用時に既読記事が表示される問題を修正

キャッシュの更新タイミングの問題で、既読にした直後に
フィルターを切り替えると既読記事が残って表示されていた。
フィルター切り替え時にキャッシュを明示的に無効化するように修正。
```

### リファクタリング

```
refactor(repository): クエリをインデックス活用に変更

全件スキャンが発生していた箇所をインデックス活用に変更し、
読み取りコストを削減。データ増加時のパフォーマンス劣化を防止。
```

### 依存関係更新

```
chore(deps): zodをv4にアップグレード

zod v4でパフォーマンス改善とバンドルサイズ削減が見込めるため更新。
破壊的変更への対応としてz.uuid()をz.guid()に置換。
```
